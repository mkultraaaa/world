Принципы работы бота

Чем бот безопасен — и как принимает решения

Бот не пытается справляться со всем подряд. Принцип: лучше задать вопрос оператору, чем сделать наугад. И когда берётся — проверяет себя на каждом шаге.

✍️
Не выдумывает
В ответе — только то, что реально произошло в админке. Никаких «активирован», если только что создал и не активирован.
✅
Проверяет результат
После каждого действия — возврат в админку и сверка, что записалось. Не записалось — эскалация.
❓
Уточняет, если двусмысленно
Несколько объектов с похожим именем — не угадывает, спрашивает. Странные значения (кап 999999, имя в 200 символов, кривой email) — блокирует.
💰
Не лезет в финансы
Любые вопросы про выплаты, балансы, комиссии — сразу к оператору. Бот не отвечает «когда выплата».
Несколько задач в одном сообщении

Бот разбирает «сделай X и забань Y» сам, без потерь

По нашим данным, около четверти сообщений в реальных чатах содержат больше одной задачи («сделай интегру и забань email», «заморозь сделку и поставь кап»). Бот выполняет все по очереди через свои защиты. Если одна задача не получилась — делает что прошло, на остаток зовёт оператора с тегом, что именно не вышло.

Создание интеграции — где бот берёт риск, а где нет

Бот включает API сам, только когда уверен на 4 проверках

Создание интеграции — самая частая операция (≈21% запросов) и самая опасная: ошибка в типе API брокера или полях кредов = клиент теряет живой трафик в кривую интеграцию. Поэтому бот берёт ответственность на себя только когда уверен.

Бот сам включает API при создании, если все 4 проверки прошли:

  • URL валидный — есть схема (https://), домен резолвится, не локалхост.
  • Тип API определён однозначно — по сигнатуре полей (token / login+password / hashed-secret).
  • Креды непустые — хотя бы один обязательный для этого типа API.
  • Гео и языки заданы — без них Vibero молча сбросит флаг live, лиды не пойдут.

Если хоть одна проверка не прошла — бот создаёт офис, но оставляет API выключенным и в ответе пишет конкретную причину («оставил выключенным: поле api_login пустое»). Финальный клик — за оператором, одну секунду, без риска кривой интеграции.

После создания работает FSM «жду подтверждения теста»: когда клиент пишет «отбили / тест прошёл / норм / деп есть» — бот сам закрывает тикет; «не работает» — эскалирует с пометкой «нужна диагностика». Закрывает ещё ~20% обращений (см. кейс #2 ниже).

3
Что замечать и записывать
Не нужно проверять «работает / не работает» — это видно по ответу. Нам важно содержание:
  • Бот ответил странно или непонятно
  • Сделал не то, что ожидалось
  • Зря эскалировал то, что мог решить сам
  • Наоборот — сделал что-то рискованное, должен был эскалировать
  • Тон не подходит — слишком сухо / робот / недостаточно уверенно / наоборот через край
4
Как вернуть фидбек Максу
Два варианта на выбор:
  • Реплай в этом же чате — на конкретный ответ бота с пометкой «фидбек: …». Контекст видно сразу, проще обсудить.
  • Личка Максу — @msoloviev, текстом или голосовым. Длинные кейсы — лучше так.
Дальше Макс по мере накопления присылает в чат сводку правок: «поправил X, уточнил Y, по Z нужно ваше решение по политике». Так лид саппорта видит, что фидбек учтён, и может сразу проверить исправление в том же чате.
5
Безопасность теста
Бот ходит в админку demo.vibero.net (демо-стенд) — не в прод-кабинеты ваших клиентов. Можно тестить любые операции, включая «забань email», «поставь кап 999999», «создай 50 офисов» — на проде никто не пострадает.

После теста

Путь от теста до клиентских чатов

После того как лид саппорта прогнал свои сценарии и фидбек учтён — заходим в боевой режим тремя шагами. На каждом — больше доверия и меньше контроля, без резких движений.

A
Shadow-режим рядом с оператором
Оператор работает как обычно, но параллельно пересылает входящие боту в отдельный чат. Видит, что бот предлагает сделать, копирует ответ если ок, правит если нет. Клиенты видят только оператора. Так оператор калибрует доверие к боту, ничем не рискуя.
риск: минимальный
B
Один пилотный клиентский чат
Выбираем чат с самым лояльным клиентом, согласовав с ним. Бот отвечает первым, оператор смотрит и может вмешаться. Все действия бота логируются — Vibero видит, что он сделал в админке. По итогам недели — корректируем.
риск: средний
C
Поэтапный rollout по чатам
Если пилот прошёл — добавляем по 3–5 чатов в неделю. Оператор в роли супервайзера, решает только нестандартные кейсы. К месяцу — основная масса рутины автоматизирована.
риск: контролируемый

#1

Создание новой интеграции

20.9% запросов ✓ выполнено
Самый частый запрос в чатах: «сделай интегру / новый офис». Бот собирает из текста имя, домен, ключи и тип API брокера, заводит офис в админке и подтверждает результат. Сразу после — открывает только что созданную карточку и сверяет, что данные действительно записались.

Что бот учёл при выполнении

  • После создания возвращается в админку и сверяет, что значения записались
  • Не активирует API сразу — оставляет это оператору для финального контроля

Обучающий вопрос → ответ из KB

13.3% запросов ✓ выполнено
«Как сделать X?» — частые обучающие вопросы. Бот ищет в knowledge base по ключевым словам, отвечает готовым текстом со ссылкой на документацию doc.vibero.tech. Бот ничего не выдумывает: текст и ссылка фиксированы в KB, оператор может проверить. Если темы нет в KB — эскалирует как раньше.
пользователь
А как мне сделать новую интеграцию с trackbox?
бот
Чтобы добавить новый офис: меню Offices → «Создать офис», вводишь имя, описание, тип API (trackbox/affboat/snp500/custom), URL и реквизиты брокера, потом подключаешь языки/гео/офферы. После создания включи флаг «Send via API» — иначе лиды не пойдут. Полная инструкция: https://doc.vibero.tech/menu____sozdat-ofis
Что бот учёл при выполнении
  • Текст ответа берётся целиком из KB — бот не генерирует новый
  • Каждый ответ содержит ссылку на полную доку — оператор может верифицировать
  • Не нашёл совпадение в KB — эскалирует, не пытается отвечать наугад

#15

Установка капа на сделку

2.8% запросов → оператору
«Поставьте кап 100 на офис X». Бот находит сделку (Deal — офис↔байер) по имени офиса, ставит указанный кап. Если сделок у офиса несколько — возвращает список ids и ждёт, чтобы клиент написал id одной (например, «238»). На уточнении бот сам перезапустит операцию с этим deal_id, не нужно переписывать всю команду.
пользователь
Поставь кап 100 на офис ACB
бот
У офиса 'ACB' 5 активных сделок (ids: [238, 1036, 910, 1028, 1026]). Напиши id той, на которую ставить кап (например, 238). @msoloviev — multiple_deals_need_clarify
Что бот учёл при выполнении
  • Кап в разумных пределах (≤10000) — иначе блок и эскалация
  • Если у офиса несколько сделок — не угадывает, спрашивает уточнить
  • После уточнения «238» автоматически продолжает выполнение, не теряет контекст
  • Гео/язык приводятся к стандартным кодам (UA, RU, NL и т.д.)
